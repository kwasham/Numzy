**Project:** Numzy – Receipt Processing Dashboard (Next.js 15, App Router, TypeScript, MUI v7, Devias Kit Pro template)

**Overview:** Implement the Phase 1 frontend features for the Numzy app. The backend is a FastAPI service with endpoints for uploading and retrieving receipts. The frontend uses Clerk for authentication and the Devias Kit Pro MUI design. Focus on building a dashboard page where users can upload receipts and view processing results.

**Requirements:**

1. **Authentication (Clerk Integration):**  
   - Ensure the app has a sign-in page using Clerk. You can use Clerk’s `<SignIn>` component or Clerk hooks to allow email/password and Google OAuth login. After successful sign-in, redirect the user to the `/dashboard`.  
   - Protect all routes under `/dashboard/*`. We have Clerk middleware set up, so unauthorized users will be redirected to sign-in. You just need to ensure the UI shows a loading state or nothing while checking auth, and possibly a sign-out button in the UI for logged-in users.
   - (You can assume `NEXT_PUBLIC_AUTH_STRATEGY` is set to `CLERK` and the Clerk provider is already wrapping the app in `app/layout.tsx`.)

2. **Receipt Upload Component:**  
   - Create a **ReceiptUploadWidget** (if not already created) that allows users to select or drag-and-drop an image or PDF file (max 10MB). Use a file input or a drag area. Validate the file type and size on the client: only accept `image/*` or `application/pdf`, with size <= 10MB.  
   - When a file is selected, call the backend `POST /receipts` endpoint to upload it. Include the user’s auth token in the request (e.g., via `Authorization: Bearer <token>` header – Clerk provides `getToken()` to get the JWT). The endpoint returns JSON of the new receipt (including its `id`, `status`, etc.).  
   - Provide user feedback: show a loading indicator (e.g., “Uploading…” on the upload button). On success, clear the selected file and show a success toast like “Uploaded: <filename>”. On error, show an error toast with a message (the backend might return an error string).  
   - After a successful upload, immediately reflect this in the UI: add the new receipt to the receipts list with status “Processing” (optimistic update). We want the user to see their uploaded receipt appear in the list right away. You can dispatch a custom browser event (e.g., `receipt:uploaded`) or simply refresh the receipts list state so that the new item is included.
   - The upload component should be integrated into the dashboard page, perhaps in a card or modal. For now, it can be a prominent section at the top of the dashboard (e.g., a card titled “Upload Receipt”).

3. **Receipts List (Dashboard Table):**  
   - Implement a component to list the user’s receipts in a table or list view. On the dashboard page (`/dashboard/receipts` route), display this list below the upload section. Use MUI Table or List components for styling. Include columns for **Receipt ID** (or file name), **Status**, **Amount** (if available), and **Uploaded Date**.  
   - Fetch the data from the backend. There is a `GET /receipts/summary` endpoint that returns recent receipts with fields: `id, filename, status, created_at, updated_at, extraction_progress, audit_progress, merchant, total, etc.`. Use SWR or React hooks to fetch this on component mount. Include the auth token in the request.  
   - Sort the list by date (newest first). If no receipts yet, show a friendly message (“No receipts uploaded yet”).  
   - For each receipt, show its status: if **COMPLETED**, you might style it with a green check icon or “Completed” label; if **PROCESSING/PENDING**, maybe a subtle spinner or “Processing…” text; if **FAILED**, show in red. You can use MUI `<Chip>` components to display status with colors.  
   - **Progress Indicators:** If a receipt is in progress (status not yet completed or failed), show a progress bar. We have `extraction_progress` and `audit_progress` (each 0–100). You can compute an overall progress (e.g., average of the two, or treat extraction as first 50%, audit as second 50%). Display a `<LinearProgress>` bar in the status column or as part of the row. If we have exact progress, use `variant="determinate"` with the percentage; if not, use `variant="indeterminate"`. Also consider showing the numeric percent (e.g., “Processing… (60%)”).  
   - Make sure the list updates in real-time or near real-time. Use SWR revalidation or a subscription: e.g., after an upload, refresh the list. Optionally, listen for server-sent events on `/receipts/events` for updates – but if that’s complex, a simple poll every few seconds for demo purposes is okay.

4. **Receipt Detail Modal:**  
   - When a user clicks on a receipt in the list, open a modal dialog to show detailed information. Use MUI’s `<Dialog>` component for the modal.  
   - Inside the modal, fetch the full details of the receipt if not already available (call `GET /receipts/{id}` for the selected receipt). This will return the extracted data and audit decision. Display a loading state (spinner or “Loading…”) while fetching.  
   - **Extracted Data Display:** Format the extracted fields in a nice way. Many fields may be present (e.g., merchant name, date, total, tax, line items, payment method). Use a two-column layout of key/value pairs. For example, use a stack of `<PropertyItem>` (or simply `<Typography>` pairs) for “Merchant: ABC Store”, “Date: 2025-10-20”, “Total: $123.45”, etc. If line-item details are available (`items` array), you can include a small table of line items (columns: item name, quantity, price). If some fields are missing, omit them or show “—”.  
   - **Audit Results:** The receipt’s `audit_decision` may include flags like `needs_audit` (boolean indicating if any rule was triggered). If `needs_audit` is true, display an alert or a section labeled “Audit Flags:” with chips for each flag. For example, if `audit_decision.amount_over_limit` is true, show an orange `<Chip label="Over Limit" />`; if `audit_decision.math_error` is true, show a red `<Chip label="Math Error" />`. If no flags (needs_audit false), you can show “No issues” or a green check icon. Include a brief tooltip or subtitle for context (e.g., “Over Limit = expense exceeds allowed threshold”).  
   - **Receipt Image Preview:** Display the actual receipt image or PDF within the modal. Use the `download_url` from the API: the backend will give a short-lived URL to the image/PDF. If it’s an image, show it in an `<img>` tag constrained to a max height (so it fits in the modal with scrolling). If it’s a PDF, you could use an `<iframe>` to embed it, or simply provide a download link (“View PDF”). Implement error handling: if the image fails to load (onError), maybe switch to an iframe or show a “Preview not available, [Download]” link.  
   - Include a close button (the “X” icon at top-right of the modal). Also consider a “Reprocess” button if `status === "failed"` – clicking it should call `POST /receipts/{id}/reprocess` then refresh the data (and close the modal or update the status to retry). This is optional for Phase 1 but good to have for completeness.

   - After implementing, test the flow: Sign in, go to dashboard, upload a receipt. Verify it appears in the list with status “Processing” and a progress bar. Simulate the backend completing the processing (you might manually set a receipt to Completed for testing or wait if connected to backend) – ensure the list item updates to “Completed” (either via a refresh or event). Implement all components in **TypeScript** and functional React style. Ensure to handle cleanup (e.g., abort fetch on unmount in detail modal, etc.) to avoid memory leaks. Use `useEffect` and `useState` as needed for data fetching and event handling. Organize the code by feature: e.g., `app/dashboard/receipts/page.tsx` for the main page, with possibly sub-components in `components/receipts/` like `ReceiptsList`, `ReceiptUploadWidget`, `ReceiptModal` for clarity. Reuse code from existing components where possible rather than duplicating.
